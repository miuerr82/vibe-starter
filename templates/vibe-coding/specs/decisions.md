# decisions.md

## 這份文件做什麼

- 用來記錄此產品專案的重要決策
- 這份文件不只記結論，也要記為什麼做這個決定
- 若之後有人質疑某個方向，應優先回來查看是否已有決策紀錄

## 不知道怎麼開始時

- 可以先從「哪些事情我們已經做了選擇，而且以後很可能會被問為什麼」開始想
- 常見內容包含：
  - 架構切分方式
  - 流程設計取捨
  - 規則限制原因
  - 外部依賴選擇原因
- 若還是不確定，可先回頭看 `0_project.md`、`boundaries.md`、`rules.md`，把那些有明顯取捨的地方整理成決策

### 可直接請 AI 協助的 Prompt

> 請根據我的 `0_project.md`、`boundaries.md`、`rules.md`、`flows.md`，幫我整理出此專案目前已經做出的重要決策。每個決策請包含「決策內容」、「原因」、「影響」，並避免寫成純進度紀錄。

## 範例

### 採用單一產品專案結構

- 決策內容：使用單一產品專案作為主要開發與交接單位
- 原因：降低初期協作複雜度，避免多 repo 管理成本
- 影響：後續文件、handoff、spec 都跟著同一個專案走

### 先定義核心行為再補細部行為

- 決策內容：`behaviors.md` 先只記錄最重要、最影響狀態的行為
- 原因：避免過早把 spec 寫得過細
- 影響：後續可按需求再擴充細部行為
